## 0. الإطار: ما تغيّر جوهرياً عن الطبقة 2 المُبطَلة

الطبقة 2 القديمة (ADR-053 §ج-1) ماتت لسببٍ بنيويّ واحد: كتبت **نيّة الإطلاق بعد حلقة `for-await`** (`claude-sdk.js:1864-1887`) — أي **downstream من موت العملية ذاتها** التي كان يُفترض أن تنجو منها. إن ماتت عملية المنسّق وسط الدور، لم يُبلَغ السطر 1877 أصلاً، فلا نيّة تُكتب، فلا صمود.

**النموذج المعتمد يقلب الترتيب:** النيّة تُكتب **لحظة الطلب الصريح للمهمة الخلفية، قبل أن يبدأ أي عمل** — من مسار طلب على مستوى التطبيق (route/WS handler)، لا من نهاية دور. المنسّق الذي يطلب المهمة **لا يُنفّذها**؛ يُسقط نيّةً ثم ينهي دوره طبيعياً (single-turn رخيص). العمل يُطلَق **طازجاً** بواسطة السوبرفايزر كخدمة systemd مستقلة، **فلم يكن قط ابناً لعملية المنسّق** — ولهذا ينجو من موتها ومن restart السيرفر.

| البُعد | الطبقة 2 المُبطَلة | النموذج المعتمد (هذه الوثيقة) |
|---|---|---|
| متى تُكتب النيّة | بعد for-await (بعد تنفيذ الورشة inline) | لحظة الطلب الصريح، قبل أي عمل |
| أين يعيش العمل | داخل عملية المنسّق (query()) | خدمة systemd مستقلة (`wf-<taskId>.service`) |
| علاقته بموت المنسّق | يموت معه | مستقل بنيوياً — لا يُلمَس |
| نقطة الحقن في المسار الحرج | `claude-sdk.js:1877` (خطر 502) | route جديد على مستوى التطبيق (لا مسّ لـ query()) |
| التسليم للمنسّق | غير موجود | مراقب دائم + حاقن دور/بطاقة |

> **توضيح المواصفة (موثَّق):** المواصفة تنصّ «تُكتب على القرص فوراً قبل موت الدور». هذا محقَّق ومُقوّى: النيّة تُكتب **قبل بدء العمل** (وهو أبكر من «قبل موت الدور»)، والدور الطالب ينتهي طبيعياً بعدها. الفرق الجوهري عن الطبقة 2: **العمل لا يجري في عملية ذلك الدور** — يُطلَق طازجاً. هذا ليس انحرافاً بل إصلاح العيب البنيوي الذي قتل الطبقة 2.

`writeLaunchIntent` القائم على `claude-sdk.js:1877` **يبقى مُبطَلاً** (المسار الميت)؛ النموذج الجديد يُعيد استخدام دالّة الكتابة نفسها لكن **من مُطلِق صريح مختلف**.

## أ. نموذج «المهمة الدائمة» (Durable Task)

### أ-1. المخطط (schema)

يُوسَّع `LaunchIntent` القائم (`server/modules/workflow-supervisor/intent.ts:14`) إلى `DurableTask`. الحقول المضافة تربط المهمة بمقصد التسليم (الشرط: {taskId، المحادثة، الرسالة، userId، المشروع، المواصفة الكاملة}):

```
DurableTask {
  schema_version: "2"
  taskId:            string   // = wfLaunchId القائم؛ charset /^[A-Za-z0-9_-]{1,128}$/ (لا حقن مسار/unit)
  userId:            integer  // من JWT المقبس — MUST be integer (fail-closed أساس العزل)
  projectPath:       string   // مسار مطلق، cwd للـ claude -p
  conversationId:    string   // [مضاف] session id المحادثة الهدف — charset صارم /^[A-Za-z0-9_-]{1,128}$/ (يُبنى منه مسار jsonl؛ لا اجتياز)
  originMessageId:   string   // [مضاف] الرسالة التي أطلقت المهمة — نفس charset taskId الصارم — للتدقيق + dedup
  spec: {                     // [مضاف] المواصفة الكاملة المُسلسَلة قبل التنفيذ
    scriptOrPrompt:  string   // نص الأمر/السكربت المُسلَّم لـ claude -p
    model?:          string|null
    effort?:         string|null
    handoffPolicy:   "card-only" | "auto-turn" | "on-demand"  // [مضاف] حوكمة الكلفة (§د)
    leafOnly:        true     // [مضاف، ثابت] الدور المحقون leaf-only دائماً (الشرط 2)
  }
  requestedAt:       string   // ISO
}
```

**لماذا مواصفة مُسلسَلة كاملة (لا أمر خام):** الطبقة 2 مرّرت `command` الخام (`claude-sdk.js:1880`) فأعادت تنفيذ الدور كاملاً (B-129). هنا المواصفة تُسلسَل صراحةً وقت الطلب مع سياسة تسليم صريحة، فلا لبس بين «نيّة» و«تنفيذ inline».

**التحقّق الصارم لكل حقول `DurableTask` — إلزامي (سدّ اجتياز المسار):** `conversationId` و`originMessageId` حقلان **ويب-المنشأ** يُبنى من `conversationId` مسار ملف فعلي `<projectDir>/<conversationId>.jsonl` (`transcript-parser.js`) للمسح (dedup) وهدف الـresume. قيمة مثل `../../../` = **اجتياز مسار للقراءة والحقن**. لذا `validateIntent` (`intent.ts:53`) الحالي — الذي يُصادق `wfLaunchId` بcharset صارم لكن **لا يعرف هذين الحقلين** — يُوسَّع في المرحلة 2 ليُصادق **كل** حقول `DurableTask` الجديدة بنفس صرامة `wfLaunchId`: `conversationId`/`originMessageId` بـ `/^[A-Za-z0-9_-]{1,128}$/`، `handoffPolicy` بقائمة قيم مغلقة، `leafOnly===true` حرفياً. أي حقل خارج القالب ⇒ رفض النيّة (fail-closed). **بوابة اختبار (المرحلة 2):** `conversationId="../../etc"` أو charset مخالف ⇒ رفض + صفر مسح/إطلاق. هذا شرط قبول صريح في §و/المرحلة 2 وجدول §ز/الشرط 1.

### أ-2. مسار التخزين

يُعاد استخدام تخطيط `supervisorStateRoot` القائم (`config.ts:62`) مع مجلد نتيجة جديد per-task:

```
<root>/                        # ~/.local/share/nassaj-dev/workflow-supervisor
├── intents/<userId>/<taskId>.json      # المُطلِق يكتب، السوبرفايزر يقرأ (0700، atomic tmp→rename)
├── tasks/<taskId>/
│   ├── supervisor.json                 # السوبرفايزر يكتب لحظة الإطلاق (unit + heartbeat + state)
│   ├── result.json.partial             # الـ task يكتب هنا أثناء التنفيذ (غير نهائي أبداً)
│   ├── result.json                     # rename ذرّي عند الاكتمال النظيف فقط
│   ├── DONE                            # sentinel: exit_code + finalizedAt (يُكتب آخر شيء)
│   ├── handoff.json                    # نتيجة معقّمة/محجّمة جاهزة للتسليم (§د/§هـ)
│   ├── pause                           # ملف تحكّم الإيقاف (lifecycle.ts القائم)
│   └── audit.log                       # سطر append-only لكل انتقال (§هـ)
├── handoffs/<conversationId>.done      # ledger exactly-once على مستوى المحادثة (لا per-task):
│                                       #   { handoffId, taskIds:[...], committedAt }  — يشمل كل
│                                       #   مهام الدفعة المدموجة (§د)، بكتابة ذرّية واحدة tmp→rename.
│                                       #   يمنع نافذة الالتزام-الجزئي التي يخلقها ledger لكل مهمة (الشرط 4).
└── supervisor.lock                     # flock مالك-واحد للمراقب الدائم (الشرط 5)
```

الأذونات: كل ما تحت `intents/`، `tasks/`، `handoffs/` بـ `0700` (سطح ويب-المنشأ). الكتابات كلها tmp→rename ذرّية (عقد ADR-RUNNER-BRIDGE-001).

**قرار ذرّية الـledger (إغلاق ملاحظة 4 — coalescing لا يكسر exactly-once):** حين يُجمَّع دور تسليم واحد لعدة مهام (§د)، **يُكتب ledger واحد ذرّي** `handoffs/<conversationId>.done` يحمل `handoffId` الدفعة و**كل** `taskIds` المدموجة معاً (tmp→rename واحد) — **لا ملف `handoff.done` لكل مهمة**. المبرر: ledger-لكل-مهمة يخلق نافذة التزام-جزئي (انهيار المراقب بعد كتابة بعض الـledgers لا كلّها ⇒ المهام الباقية تبدو غير-مُسلَّمة ⇒ إعادة حقن ⇒ تسليم مزدوج). الكتابة الذرّية الواحدة تجعل الدفعة كلها «مُسلَّمة» أو «غير مُسلَّمة» بلا حالة وسطى.

### أ-3. دورة الحياة وحالات النهاية

```
[requested] ──(المُطلِق يكتب intent)──▶ [pending]
[pending]   ──(GATE2 + concurrency + launchScope)──▶ [running]   (supervisor.json لحظة الإطلاق)
[running]   ──(الـ task يكتب result.json.partial → rename → DONE)──▶ نهاية المنتِج
                                    │
        ┌───────────────────────────┼───────────────────────────┐
        ▼                           ▼                           ▼
   exit_code=0                  exit_code≠0                 قُتل (kill/timeout)
   result.json موجود+DONE       result.json ناقص/DONE       لا DONE أو DONE بإشارة قتل
        ▼                           ▼                           ▼
   [SUCCEEDED]                  [PARTIAL]                   [CRASHED]
        └───────────────────────────┴───────────────────────────┘
                                    │ (المراقب يلتقط، يبني handoff.json معقّماً)
                                    ▼
                          [handoff-pending] ──(بطاقة Tier-A أو دور Tier-B)──▶ [DELIVERED]
                                    │                                              │
                          (بلغ الميزانية/العلم card-only)                    (handoff.done يُكتب)
                                    ▼
                          [DELIVERED-card-only]
```

حالات النهاية الثلاث للمنتِج (**تمييز إلزامي — الشرط 4 وبلوكر qa-critic #2**): `SUCCEEDED` / `PARTIAL` / `CRASHED`. لا تُخلط في «is-active» مستوى-إشارة كما في السوبرفايزر الحالي (`systemd.ts:200-206` يشغّل `claude -p` **بلا ملف نتيجة/output-format** — هذه الفجوة الأولى التي يسدّها التصميم).

**قاعدة التوفيق عند غياب `DONE` (إلزامية — تسدّ السيناريو الذي لا يملكه المخطط أعلاه):** يوجد سيناريو رابع لا يُغطّيه «SUCCEEDED يشترط DONE»: claude خرج **نظيفاً (0)**، الغلاف نفّذ `rename(partial→result.json)`، **ثم** `kill -9`/`RuntimeMaxSec`/إقلاع أصابه **قبل كتابة `DONE`**. الحصيلة على القرص: `result.json` **مكتمل موجود**، `DONE` **غائب**، والوحدة `inactive` أو `failed`. قاعدة «لا ثقة دون DONE» وحدها تترك المهمة إمّا **عالقة أبداً** في `handoff-pending` (تسريب)، أو مُصنَّفة `CRASHED` رغم نجاحها (تحت-تسليم). **القاعدة الحاسمة:**

> إذا كانت is-active **طرفية** (`inactive`/`failed`) **و** `DONE` غائب **بعد مهلة سماح** (`RECONCILE_GRACE_MS`، تقديراً 10s بعد أول رصد طرفية) ⇒ يُحسم التصنيف من **is-active + وجود `result.json`**، لا من انتظار `DONE`:
> - `failed` (أو لا `result.json`) ⇒ **CRASHED** (يُعرَض، لا يُسلَّم كامل).
> - `inactive` **و** `result.json` موجود مكتمل ⇒ **PARTIAL-untrusted**: النتيجة موجودة لكن **افتقاد `DONE` يعني الغلاف لم يختم**، فتُعامَل معاملة الجزئي (تُعرَض للمستخدم مع وسم «غير مختومة»، **لا حقن دور LLM تلقائي عليها**)؛ لا تُسلَّم كنتيجة `SUCCEEDED` كاملة.

المهلة تمنع سباق «is-active صارت inactive قبل أن يكتب الغلاف DONE بلحظات». هذا يُبقي التصنيف **حاسماً بلا تعليق أبدي وبلا ادّعاء اكتمال لا يملكه المراقب** — وهو بند قبول صريح في T-819 (§و/المرحلة 1).

### أ-4. ضمانات الذرّية وexactly-once + تحليل الأعطال

**بروتوكول المنتِج (الـ task):** يُطلَق `claude -p ... --output-format json` عبر غلاف صغير (wrapper) يُنفّذ:
1. `claude -p '<spec>' --output-format json > result.json.partial` (كتابة تدريجية إلى `.partial` حصراً).
2. عند exit النظيف: `rename(result.json.partial → result.json)` (ذرّي على نفس الـ FS).
3. ثم كتابة `DONE` أخيراً: `{exit_code, signal, finalizedAt}` (tmp→rename).
الترتيب حاسم: **`result.json` لا يوجد إلا مكتملاً؛ `DONE` هو آخر أثر.** المراقب لا يثق بشيء دون `DONE`.

**بروتوكول المستهلك (المراقب):** exactly-once على التسليم عبر ledger محتوى-معنوَن:
1. الدور/البطاقة المحقونة تحمل `handoffId = hash(taskId)` حتمياً.
2. قبل الحقن: المراقب يفحص ذيل `<conversationId>.jsonl` عن رسالة تحمل `handoffId` — إن وُجدت ⇒ تخطّى (سبق التسليم).
3. بعد التزام الحقن في jsonl: كتابة `handoff.done` ذرّياً.

**تحليل الأعطال (مطلوب صراحةً):**

| العطل | السلوك | الضمان |
|---|---|---|
| **kill -9 أثناء كتابة النتيجة** | لا rename حدث ⇒ يوجد `.partial` فقط، لا `result.json` ولا `DONE` | المراقب يصنّف `CRASHED`/`PARTIAL` (يعرض الجزئي + تدقيق)، **لا نتيجة كاملة ممزّقة تُسلَّم قط**. |
| **restart أثناء التنفيذ** | `wf-<taskId>.service` يملكه مدير systemd للمستخدم، ينجو restart nassaj-dev وموت المراقب | عند إقلاع المراقب: يعيد التعداد بـ `listActiveUserScopes` + يقرأ `tasks/*/supervisor.json` ⇒ يعيد الربط. **صفر عمل ضائع.** |
| **restart بعد الاكتمال وقبل التسليم (DONE موجود)** | `result.json`+`DONE` على القرص؛ لا سطر في ledger المحادثة | المراقب يعيد المحاولة؛ `handoffId` يمنع الازدواج. |
| **نجاح ثم kill-9/إقلاع قبل كتابة `DONE`** | `result.json` مكتمل، `DONE` **غائب**، is-active طرفية | **قاعدة التوفيق (§أ-3):** بعد مهلة سماح ⇒ PARTIAL-untrusted (يُعرَض، لا يُسلَّم كامل، لا حقن تلقائي) أو CRASHED إن `failed`. **لا تعليق أبدي ولا ادّعاء اكتمال لا يملكه المراقب.** |
| **restart بعد حقن الدور وقبل كتابة الledger** | الرسالة في jsonl لكن ledger المحادثة لم يُكتب | فحص ذيل jsonl عن `handoffId` يجده ⇒ يكتب ledger الدفعة الذرّي فقط، **لا حقن ثانٍ**. |
| **coalescing: انهيار بعد التزام دور مدموج قبل الledger** | دور واحد لعدة مهام في jsonl، ledger الدفعة غائب | **ledger ذرّي متعدّد-المهام (§أ-2):** الدفعة كلها «مُسلَّمة» أو لا — لا حالة وسطى ولا تسليم مزدوج لبعض مهام الدفعة. |
| **المراقب يقرأ نتيجة جزئية** | مستحيل بناءً — يفتح على `DONE`+`result.json` أو قاعدة التوفيق الطرفية، لا `.partial` أبداً | تمييز `SUCCEEDED`/`PARTIAL`/`CRASHED` قطعي. |
| **سطر `handoffId` نصف-مكتوب في jsonl** (append منقطع بموت عملية resume) | المطابقة قد تُخطئ handoffId جزئي ⇒ خطر إعادة حقن | **الحسم في T-819 لا الافتراض:** المطابقة على سطر jsonl **مُحلَّل بالكامل** (JSON.parse سليم) لا regex نصّي؛ سطر غير قابل للتحليل = «لم يُسلَّم بعد» تحفّظاً + إعادة محاولة idempotent عبر ledger الدفعة. مُختبَر على jsonl ممزّق حقيقي (§و/المرحلة 1، بند المستهلك). |
| **tmp نصف-مكتوب للنيّة** | rename ذرّي يمنع القراءة الجزئية؛ ملف `.tmp-` يُتخطّى بالاسم (`supervisor.ts:132`) | السوبرفايزر لا يرى نيّة نصف-مكتوبة. |

هذا هو **جوهر T-819** (المرحلة 1): إثبات الجدول أعلاه على مخرَج `claude -p` **حقيقي** لا مصطنع — **على جانبَي المنتِج (التقاط النتيجة) والمستهلك (dedup التسليم على jsonl ممزّق)** معاً (درس reconcile 6.5% — `feedback_synthetic_fixtures_false_confidence` — طابق 6.5% من الواقع رغم اختبارات خضراء على fixtures مصطنعة؛ الدرس يُطبَّق بصرامة متساوية على المستهلك).

## ب. المكوّنات وتكاملها مع الموجود

### ب-0. قرار معماري: daemon واحد للتحكّم + خدمة عابرة لكل مهمة (كلاهما، بطبقتين)

السؤال المطروح (template-unit لكل مهمة أم daemon واحد) جوابه **الاثنان معاً في طبقتين مختلفتين، بمبرر:**

- **مستوى التحكّم = daemon واحد دائم** (`workflow-supervisor.service`، القالب مبنيّ). يحمل الـflock (مالك-واحد، الشرط 5)، يستطلع النيّات، يُطلق، **يراقب الاكتمال، ويُسلّم**. **المبرر:** flock واحد = رؤية عالمية للتزامن + coalescing + منع مراقبين متوازيين. daemon-لكل-مهمة يُضاعف الـflocks ويفقد المنظور العالمي.
- **مستوى الحمل = خدمة systemd عابرة لكل مهمة** (`wf-<taskId>.service`، `systemd.ts:158` مبنيّ). كل مهمة: `MemoryMax`/`RuntimeMaxSec` مستقل، دورة حياة مستقلة (تنجو من موت الجميع)، حياة is-active مستقلة. **المبرر:** daemon واحد يشغّل كل المهام يقرن دورات حياتها ويفقد سقف الموارد per-task.

**الخلاصة:** التحكّم مركزيّ (daemon)، الحمل معزول (خدمة عابرة per-task). هذا التقسيم مبنيّ أصلاً في الوحدة.

### ب-2. المراقب الدائم (التوسعة الجوهرية)

الحلقة القائمة (`supervisor.ts:147`) تُطلق فقط. تُوسَّع إلى دورة كاملة:

```
runSupervisor():
  1. acquire flock(supervisor.lock, non-block)  → إن فشل: مثيل آخر يعمل، اخرج بهدوء (الشرط 5)
  2. reconcile-on-boot: تعداد wf-*.service الحيّة + قراءة tasks/*/ ⇒ إعادة ربط المهام الناجية
  3. loop:
       a. pollIntents()      → GATE2 → concurrency → launchScope (كما القائم)
       b. monitorRunning()   → لكل مهمة running: is-active + فحص DONE
       c. onTerminal(task)   → التقاط النتيجة → build handoff.json معقّم → تسليم (Tier A/B)
       d. sweepCompleted()   → تنظيف tasks/ القديمة (timer)
```

الـflock بـ `flock(2)` على `supervisor.lock` (نمط `runner` القائم يستعمل flock per-project — يُعمَّم). المراقب خفيف (`MemoryMax=256M` في القالب)، والمهام تحمل سقفها.

## ج. التدفقات الحرجة

### ج-1. المسار السعيد (نجاح + تسليم)
1. المستخدم يطلب صراحةً «شغّل بالخلفية» في محادثة → route الإطلاق يكتب `DurableTask` (conversationId = المحادثة الحالية) بـ`0700` ذرّياً. الدور الطالب ينتهي (رخيص). الشارة: «1 مهمة خلفية».
2. المراقب: GATE2 PASS → concurrency PASS → `launchScope` (خدمة عابرة، `--output-format json`، `--setenv CLAUDE_CONFIG_DIR` معزول). `supervisor.json` لحظة الإطلاق.
3. المهمة تكتب `result.json.partial` → عند exit=0: rename → `result.json` → `DONE`.
4. المراقب يرى `DONE` + is-active=inactive ⇒ `SUCCEEDED`. يبني `handoff.json` معقّماً/محجّماً.
5. **Tier-A افتراضياً:** بطاقة `task-notification` تظهر في المحادثة (بلا دور LLM): «اكتملت المهمة الخلفية — سلّمها للمنسّق». `handoff.done` يُكتب.
6. **Tier-B (طلب صريح أو `handoffPolicy:auto-turn`):** المراقب يأخذ قفل الدردشة، يحقن دور `claude -p --resume <conversationId>` بأدوات leaf-only حاملاً النتيجة المعقّمة، ثم `handoff.done`. المنسّق يستهلك النتيجة ويكمل.

### ج-2. جزئي/انهيار
- المهمة تخرج بـ exit≠0 أو تُقتل: `result.json` ناقص/غائب. المراقب يصنّف `PARTIAL`/`CRASHED` من `DONE`(إن وُجد)+is-active(`failed`). البطاقة تعرض «المهمة الخلفية لم تكتمل (جزئي/انهيار)» + مؤشّر للأثر الجزئي على القرص. **لا دور LLM على نتيجة منهارة** (لا تُحرَق حصة على فشل).

### ج-3. restart وسط التنفيذ
- restart nassaj-dev: المهام خدمات systemd مستقلة ⇒ تكمل. المراقب (وحدة user منفصلة) قد يُعاد تشغيله بـ`Restart=on-failure` ⇒ reconcile-on-boot يعيد الربط. **صفر عمل ضائع، صفر تسليم مزدوج** (handoffId + handoff.done).

### ج-4. سباق الحقن مع دور حيّ (الشرط 3 — الأخطر على السلامة)
- **الخطر:** المراقب يحقن `claude -p --resume <sid>` بينما دور دردشة حيّ يكتب نفس `<sid>.jsonl` ⇒ عمليتا CLI تُلحقان ⇒ إفساد jsonl.
- **الحل (تسلسل يشمل الدردشة الحيّة):** قفل استشاري per-conversation (`flock` على `<sid>.lock`) يأخذه **الطرفان**: مسار الدردشة الحيّ في `claude-sdk.js` (قبل query()، يُحرَّر بعد for-await) **والحاقن**. **هذا المسّ الوحيد للمسار الحرج**، خلف العلم (OFF ⇒ لا قفل ⇒ سلوك byte-identical)، ويتطلّب فيتو qa-critic + tester (المرحلة 4).

**دلالة القفل الكاملة (محدَّدة — تعالج الاتجاهين لا اتجاهاً واحداً):**
1. **الطرفان يأخذانه غير-حاجب** (`flock LOCK_EX|LOCK_NB`) — لا انتظار أعمى.
2. **أولوية الإنسان مطلقة:** الحاقن يفشل **فوراً** إن كان القفل مأخوذاً (دور بشري يحمله) ⇒ يؤجّل (queue) ويعيد لاحقاً. الدور البشري **لا يُحجب بالحاقن في الحالة الشائعة** (الحاقن لا يبدأ أصلاً).
3. **سقف حجب دور الحاقن (يعالج «الإنسان محجوب بالحاقن» — الفجوة أ):** حين **يملك** الحاقن القفل، دوره **leaf-only محدود بمهلة قصوى `HANDOFF_MAX_HOLD_MS` (تقديراً 90s)** — أقصر من دور كامل لأنه بلا subagents/أدوات توليد (leaf-only، §هـ-2): مجرّد استهلاك نتيجة **معقّمة محجّمة (≤32KB)** + رد قصير. عند تجاوز المهلة ⇒ **يُقاطَع الحقن ويُحرَّر القفل فوراً** (المهمة تبقى handoff-pending، تُعاد لاحقاً — لا تُفقد). فلا يحمل الحاقن القفل «دقائق مفتوحة».
4. **مسار الدردشة الحيّ حين يجد القفل مأخوذاً (نادر: حقن بدأ لتوّه):** ينتظر بمهلة قصيرة مقيّدة `CHAT_LOCK_WAIT_MS` (بحدود سقف حجب الحاقن) مع مؤشّر واجهة «تسليم خلفي جارٍ، لحظة»؛ إن لم يُحرَّر خلالها ⇒ يُبلَّغ المستخدم ويُعاد إرسال الطلب — **لا حجب لا-نهائي على المسار الحرج بأي حال**.
5. **البديل الأخفّ المرفوض:** idle-check (`isClaudeSDKSessionActive`, `claude-sdk.js:~2125`) بلا قفل — يمنع الازدواج (handoffId) لكن **لا يمنع إفساد jsonl من إلحاق متزامن** (ثقب TOCTOU). الشرط 3 يطلب تسلسلاً يشمل الدردشة الحيّة صراحةً ⇒ القفل هو الجواب الأمين. يُوثَّق البديل للشفافية لا كحلّ.

> **تفاعل القفل مع الدرين (الفجوة ج — يُسمّى صراحةً):** القفل ملف flock عبر-العمليات، **لا** يُدخل جلسة في `activeSessions` ولا يمسّ `getDrainBlockingClaudeSessions`/`session.detached` مباشرة. الخطر الدرينيّ الوحيد: **مسار دردشة حيّ محجوب على القفل قد يُطيل جلسة نشطة يعدّها الدرين** أثناء restart. التخفيف: سقف الانتظار (البند 4) يقيّد الحجب بـ`CHAT_LOCK_WAIT_MS`، فلا إطالة غير محدودة. **هذا الخطر مرفوع صراحةً في §ح-3 (لا يُطوى تحت «لا اقتران درين» في §ج-6).**

### ج-5. بلوغ سقف التزامن
- `admitLaunch` يرفض N+1 (per-user) ⇒ النيّة تبقى على القرص (`queued`)، تُعاد لاحقاً عند تحرّر سعة (`supervisor.ts:95`). سقف عالمي يُضاف (host-wide) لمنع OOM (درس 2026-06-06). **بوابة اختبار:** N+1 عند السقف ⇒ رفض/تأجيل مرئي، **لا نوبة OOM**.

### ج-6. تفاعل الدرين (safe-restart، B-95) والذاكرة (B-130)
- **المهام** مستقلة عن عدّ الدرين (يعدّ أبناء claude لعملية nassaj-dev؛ المهام أحفاد systemd). **لا تُدخِل المهام في الدرين** — تنجو منه بالتصميم. `safe-restart.sh` (B-95) يُوسَّع بـ**رؤية قراءة-فقط** لـ`wf-*.service` الحيّة (المالك يراها قبل restart)، دون تأجيل restart لها. B-130 (أُلغي `max_memory_restart`) يعني لا restart ذاكرة يقتل المراقب؛ المراقب بـ`MemoryMax=256M` مستقل. **المهام: لا اقتران درين جديد.**
- **تصحيح مهم:** الاقتران الدرينيّ الحقيقي ليس المهام بل **قفل مسار الدردشة** (الشرط 3، §ج-4): جلسة حيّة محجوبة على القفل أثناء restart. مقيّد بسقف `CHAT_LOCK_WAIT_MS` ومرفوع صراحةً كمخاطرة في §ح-3 — لا يُطوى هنا.

## د. حوكمة الكلفة

**المشكلة (الشرط 6):** كل تسليم Tier-B = دور كامل ~73k توكن على حصة المستخدم. تسليم كل مهمة بدور = حرق حصة.

**مساران بسياسة `handoffPolicy`:**

| السياسة | الآلية | الكلفة | متى |
|---|---|---|---|
| `card-only` (افتراضي) | بطاقة `task-notification` non-LLM (نمط `buildReconcileMessage`, `workflow-reconcile.service.ts:344`) تُشتقّ وقت `getSessionMessages` من `tasks/<id>/handoff.json` | **~0 توكن** | كل مهمة، دائماً — الرؤية الأساسية |
| `on-demand` | البطاقة تعرض زر «سلّم للمنسّق»؛ الدور يُحقن فقط بضغط المستخدم | ~73k عند الطلب فقط | حين قد يحتاج المنسّق التفاعل |
| `auto-turn` | حقن تلقائي بعد الاكتمال | ~73k/تسليم | مهام يجب أن يكملها المنسّق بلا تدخّل |

**Coalescing (إلزامي لـauto-turn/on-demand):** عدة مهام لنفس `conversationId` تكتمل ضمن نافذة (مثلاً 60s) تُجمَّع في **دور واحد** يحمل كل نتائجها المعقّمة. المراقب يحمل طابور تسليم per-conversation ويطلق دوراً واحداً عند إغلاق النافذة أو بلوغ عتبة عدد. **الالتزام الذرّي:** الدفعة المدموجة تُختَم بـ**ledger محادثة ذرّي واحد** (`handoffs/<conversationId>.done`، §أ-2) يحمل كل `taskIds` الدفعة — لا ledger لكل مهمة (وإلا نافذة التزام-جزئي ⇒ تسليم مزدوج، الشرط 4).

**نموذج الكلفة الملموس (للمالك — الأرقام تقديرية مقترحة، يثبّتها قرار المالك في §ح):**

| البند | التقدير | ملاحظة |
|---|---|---|
| تسليم Tier-B واحد (دور محقون) | **~73k توكن** | رقم مرجعي (فيتو qa-critic/سجل المالك) — دور كامل على حصة المستخدم |
| دفعة مدموجة (N مهام، coalescing) | ~73k + (N−1)×~8k | كل نتيجة إضافية معقّمة محجّمة ≤32KB ≈ ~8k توكن؛ دفعة 3 ≈ **89k** لا 219k |
| `HANDOFF_TOKENS_PER_CONVERSATION_MAX` (مقترح) | **~300k توكن/يوم** | ≈ 3–4 تسليمات مدموجة للمحادثة الواحدة |
| `HANDOFF_TOKENS_PER_USER_MAX` (مقترح) | **~1M توكن/يوم** | عبر كل محادثات المستخدم |
| **أسوأ حالة (auto-turn بلا coalescing)** | **3×73k ≈ 219k دفعة واحدة** | مستخدم عند سقف التزامن (3 مهام per-user) تكتمل متفرّقة؛ يُحدّ بالسقف اليومي per-user + السقف العالمي |

**السياق الحرج:** خطة Max = جلسة 5 ساعات. تسليم واحد ~73k **ليس هيّناً** على ميزانية جلسة. لذا `card-only` افتراضاً + coalescing + السقف اليومي **ضرورات لا رفاهية** — بدونها، تسليمات auto-turn تحرق حصة المنسّق (نفس نمط 92697edb الذي أشعل B-103).

**ميزانية (الإنفاذ):** سقف per-user وper-conversation (الجدول أعلاه، حقول config). التجاوز ⇒ هبوط قسري إلى `card-only` (لا دور)، مع بطاقة «تجاوزت ميزانية التسليم — النتيجة على القرص».

**مسار العرض غير-LLM هو الافتراضي، لا الاستثناء.** الدور المحقون (المكلف) استثناء صريح محكوم بالسياسة + الميزانية + coalescing.

> **توضيح المواصفة (موثَّق، محكوم بالشرط 6) — والاختيار الافتراضي قرار مالك حاجز:** المواصفة تقول «مراقب يحقن دوراً جديداً يسلّم المخرجات **للمنسّق ليكمل**». التصميم يجعل الدور المحقون **أحد مسارين**: الافتراضي بطاقة non-LLM (~0 توكن)، والدور LLM محجوز لـon-demand/auto-turn مع coalescing وميزانية. **آلية** المستويين ضمن تفويض الشرط 6 (دور كامل ~73k لكل تسليم غير محتمَل اقتصادياً). **لكن اختيار المستوى الافتراضي قرار منتج للمالك، لا حاشية:** `card-only` افتراضاً يعني أن المنسّق **لا يكمل تلقائياً** — البطاقة تُشعر الإنسان ليقرّر؛ وهذا تحوّل دلالي عن نصّ المالك «ليكمل» (الذي يقتضي `auto-turn`). **يُرفع كقرار مالك صريح في §ح-2 (المستوى الافتراضي)** — لا يُحسَم هنا.

## هـ. الأمن والامتثال

### هـ-1. fail-closed على userId (الشرط 1 — حاجز ToS)
- **التأطير الدقيق (تصحيح):** `resolveProviderEnv` fail-OPEN (`resolve-provider-env.js:74-77`): `userId===null/undefined/''` ⇒ بيئة أساس **بلا `CLAUDE_CONFIG_DIR`** = اشتراك المالك (مشاركة صامتة، مخالفة ToS). لكنه **ليس «الجذر» في مسار الإطلاق** — إنه **سطح خطر يُحيَّد أصلاً هناك بـGATE2**: `authorizeLaunch` (`ownership-guard.ts:78,89,99`) يرفض كل userId غير-integer/غير-مالك **قبل** أي بيئة تُبنى، والبيئة تُبنى فقط لـALLOW (بمعرّف integer مُصادَق). فمسار الإطلاق مغلق سلفاً.
- **أين يلزم الغلاف الصارم فعلاً (لا تعميم):** المساران اللذان قد يستدعيان `resolveProviderEnv` **خارج** `authorizeLaunch` هما **الحاقن (`handoff-injector`, resume spawn) والتقاط النتيجة (إن أعاد بناء بيئة)** — هذان لا يمرّان بالضرورة عبر GATE2 لكل spawn، فيلزمهما **حتماً** `resolveProviderEnvStrict(userId, provider)`: يرمي (throw) إن لم يكن `userId` integer موجباً، **يُستدعى صراحةً في هذين المسارين تحديداً**. **لا يُعدَّل `resolveProviderEnv` المشترك** (null=system دلالة شرعية لمستدعين آخرين).
- **عزل userId في الحاقن (إلزامي — فجوة التقرير §5-4):** الحاقن **يشتقّ `userId` من `DurableTask` المُصادَق بـGATE2 ويُعيد فحص `Number.isInteger` قبل أي `resume` spawn** — **لا يثق بحقل على القرص قد يُفسَد**؛ `resolveProviderEnvStrict` هو البوابة قبل الـspawn.
- **البوابة الحيّة (GATE2 المُوسَّعة، المرحلة 2):** `userId=null/"abc"/999-غير-مالك` ⇒ رفض صريح + **صفر إطلاق** + **لا حصة مالك/نظام تُلمَس** (تحقّق `lsof`/`/proc/environ` أن اعتماد المالك لم يُفتَح). العزل عبر `--setenv` مُثبَت PASS مسبقاً (GATE1). GATE2 مبنيّ ومُختبَر (gate2-ownership 6/6).

### هـ-2. الدور المحقون leaf-only (الشرط 2)
- الدور المحقون (Tier-B) يُطلَق بسياسة أدوات **تعطّل كل أدوات التوليد الخلفي**: `Task`, `Workflow`, تفريخ عمليات خلفية. `spec.leafOnly:true` ثابت. **المبرر:** دور محقون يفرّخ مهمة خلفية = B-103 مُنقول (الدور المحقون single-turn، إن أطلق خلفياً مات ثانية). التطبيق: `disallowedTools` صريحة في `sdkOptions` للحاقن + تأكيد بالاختبار أن الحقن لا يكتب نيّة جديدة.

### هـ-3. النتيجة المحقونة = بيانات غير موثوقة (الشرط 4)
- نتيجة المهمة = مخرَج نموذج = **غير موثوقة**. قبل دخولها أي تسليم:
  - **تعقيم:** تجريد/تهريب محارف التحكّم؛ تغليف صريح `<background_task_result untrusted="true">…</background_task_result>` في الدور المحقون فيعامله المنسّق **بيانات لا تعليمات** (ضد حقن prompt المضخّم على حصة المستخدم).
  - **تحجيم:** قصّ إلى ميزانية بايت (16–32KB) + مؤشّر «مقصوص، الكامل في `tasks/<id>/result.json`».
  - **ميزانية:** §د.
  - **قتل:** كل مهمة قابلة للإيقاف (`stopScopeWorkflow`, `lifecycle.ts:38`) عبر route مُصرَّح + الشارة.
  - **تدقيق:** `audit.log` سطر append-only لكل انتقال (taskId, userId, projectPath, bytes, tokens, outcome).

### هـ-4. سطح الحقن الويب
- النيّة ويب-المنشأ ⇒ `validateIntent` (`intent.ts:53`) صارم (integer userId، charset، مسار مطلق)؛ الأذونات `0700`؛ المُطلِق يكتب نيّة فقط **لا يُطلق شيئاً مميّزاً** (حدّ الثقة خارج nassaj-dev — القرار ب).

## و. الخطة المرحلية

> الترقيم من T-820 صعوداً **اقتراح** — المنسّق يثبّته باللوحة. المرحلة 1 = T-819 (معتمد).

### المرحلة 1 — T-819 (معتمد): spike الإثبات [حاجزة]
**النطاق:** إثبات القطعتين غير المُثبتتين على مخرَج `claude -p` **حقيقي** (لا مصطنع):
- (أ) التقاط نتيجة scope ذرّي + exactly-once: تصنيف نجاح/جزئي/انهيار + idempotency تحت kill -9 أثناء الكتابة.
- (ب) مراقب دائم ينجو restart بصفر حقن مزدوج/ضائع + flock مالك-واحد.

**معيار القبول (قابل للقياس) — جانبا المنتِج والمستهلك بصرامة متساوية:**

*جانب المنتِج (التقاط النتيجة):*
1. ≥20 تشغيل `claude -p --output-format json` فعلي تحت `systemd-run` scope يغطّي الأصناف الثلاثة (نجاح نظيف، جزئي/timeout، انهيار kill -9 عند إزاحات بايت **عشوائية** أثناء كتابة النتيجة). **الـfixtures = stdout+exit الفعلي المُلتقَط، لا مُصطنع.**
2. المصنّف يعطي الصنف الصحيح 100% من الـ20+؛ kill -9 أثناء الكتابة **لا يُنتج `result.json` ممزّقاً قط** (فقط `.partial`) — مُتحقَّق بحقن القتل عند 100 إزاحة عشوائية.
3. **قاعدة التوفيق عند غياب `DONE` (بند قبول صريح — ثقب 2-أ):** kill -9 **بين `rename` و`DONE`** على تشغيل فعلي ⇒ قاعدة §أ-3 تُنتج تصنيفاً **حاسماً** (PARTIAL-untrusted أو CRASHED) بعد مهلة السماح — **لا تعليق أبدي، ولا ادّعاء SUCCEEDED كامل**. مقيساً على ≥10 تشغيلات مُقاطَعة في هذه النافذة تحديداً.

*جانب المستهلك (dedup التسليم) — الدرس مطبَّق بصرامة متساوية:*
4. idempotency: تكرار خطوة الـfinalize N مرّة على نفس مجلد المهمة ⇒ أثر تسليم **واحد** (handoffId).
5. **dedup التسليم على jsonl حقيقي مع سطر `handoffId` ممزّق/جزئي (بند قبول صريح — ثقب 2-ب، نظير حادثة 6.5%):** حقن سطر `handoffId` **نصف-مكتوب/غير قابل للتحليل** في `<conversationId>.jsonl` فعلي ⇒ المطابقة (JSON.parse سليم لا regex) تعامله «لم يُسلَّم» تحفّظاً + إعادة محاولة idempotent عبر ledger الدفعة ⇒ **صفر تسليم مزدوج، صفر ضائع**. **«صفر مزدوج/ضائع على jsonl سليم» وحده لا يكفي** (كما مرّ reconcile 18/18 وفشل ميدانياً).
6. صمود المراقب: قتل المراقب (kill -9) وسط تشغيل حقيقي ثم إقلاعه ⇒ يعيد الربط (يجد `wf-*.service` + السجل القرصي) ويسلّم **مرّة واحدة** (صفر مزدوج/ضائع) عبر ≥10 محاولات restart-عند-إزاحة-عشوائية.
7. flock: مثيلا مراقب متزامنان ⇒ واحد فقط يعمل، الآخر يخرج بهدوء.

*تحقّق ميداني:*
8. **soak ميداني:** دورة كاملة إطلاق→اكتمال→تسليم حيّاً على nassaj-dev لـ**≥10 مهام موزّعة على الأصناف الثلاثة** (لا «≥N» مبهم)، تحقّق على **transcript فعلي** (لا fixtures) أن البطاقة/الدور يظهر ويُنسَب `task-notification` (لا للمستخدم).

**المخاطر:** بناء التقاط النتيجة **والتسليم** من الصفر فوق نمط فشل بالمشروع (سابقة reconcile 6.5%) — يخفّفه معيار الـfixtures الحقيقية + بند jsonl الممزّق + soak. **الاعتماديات:** عزل CLAUDE_CONFIG_DIR (GATE1 PASS). **بوابة:** فشل أيٍّ ⇒ إسقاط النموذج إلى رؤية غير-LLM أو CCR (عنوان T-819).

## ح. المخاطر المفتوحة والأسئلة للمالك

1. **[أسئلة نطاق] فئة حوادث B-103 الفعلية لا يغطّيها النموذج تلقائياً.** الحوادث التي أحرقت الحصة (92697edb: مراجعات وتشخيص وسط الدور) هي عمليات `claude -p` من الharness/قرار وسط الدور، **لا** مهام مُطلقة صراحةً. النموذج يعالجها **فقط إن أطلقها المستخدم/المنسّق صراحةً كمهام خلفية** (الشرط 8: أمانة النطاق). **سؤال المالك:** هل يكفي هذا، أم يُطلب حاجز harness منفصل (رد ممسوك/منع الانفصال الخلفي) بالتوازي كما أوصى qa-critic؟ (الافتراضي الآمن — رد ممسوك — يبقى المسار العملي حتى التفعيل.)

2. **[قرار حاجز] المستوى الافتراضي للتسليم: card-only مقابل auto-turn.** الآلية بمستويين مقرّة ضمن الشرط 6 (§د): `card-only` (بطاقة non-LLM ~0 توكن؛ الإنسان يقرّر متى يُحقن دور) يحوّل دلالة المواصفة «المنسّق ليكمل» إلى «الإنسان يقرّر»؛ `auto-turn` يطابق نص المواصفة حرفياً لكن بكلفة ~73k/تسليم من حصة الاشتراك (نموذج الكلفة في §د). التوصية المعمارية: `card-only` افتراضاً مع إتاحة `auto-turn` لكل مهمة عبر `handoffPolicy` — والحسم للمالك.

3. **[مخاطرة] المسّ الوحيد للمسار الحرج (قفل الدردشة، المرحلة 4).** أي تعديل في `claude-sdk.js` خطر 502 تاريخي (B-95/B-N-DRAIN). التخفيف: خلف العلم (OFF=byte-identical) + فيتو qa-critic + tester + مراجعة من طرفية SSH مستقلّة. **سؤال:** هل يقبل المالك مبدئياً مسّاً مُقيَّداً بالمسار الحرج للمرحلة 4، أم يُفضَّل النموذج المقيَّد بـidle-check (يترك ثقب TOCTOU على إفساد jsonl، أضعف على الشرط 3)؟

4. **[مخاطرة تقنية] التقاط النتيجة من الصفر فوق نمط فشل بالمشروع** (سابقة reconcile 6.5% رغم اختبارات خضراء). مُخفَّف ببوابة T-819 (fixtures حقيقية + soak ميداني) — لكنه يبقى أكبر مجهول تقني. **هذا سبب كون T-819 حاجزاً قبل أي تصميم كامل يُنفَّذ.**

5. **[قرار مؤجَّل] CCR remote (`remote_launched`) كبديل جذري.** إن فشل T-819، النموذج يهبط إلى رؤية غير-LLM أو CCR (M-BG-3، بحث مؤجَّل: ToS/خصوصية/تكلفة). لم يُقيَّم للإنتاج بعد.

6. **[دلالات reboot، مُحسَمة سلفاً]** المهام العابرة لا تنجو reboot ولا تُستأنف تلقائياً ⇒ يتيمة-مرئية + استئناف يدوي (اتساقاً مع رفض resumeFromRunId الآلي same-session). مُثبَّتة في §ج/المرحلة 5.

7. **[أثر ToS، محسوم لا يُعاد فتحه]** العزل per-user (`CLAUDE_CONFIG_DIR`) إلزامي؛ fail-open في `resolveProviderEnv` مسدود بالغلاف الصارم + GATE2. أي كسر هنا = مشاركة اشتراك صامتة.
