# ADR-060: لجنة تيتّم الورشات الخلفية (B-103)
## إعادة تأطير — الديمومة تعالج سطح المنتج لا ألم المنسّق

**التاريخ:** 2026-07-15  
**الحالة:** مقبول  
**المسؤولون:** architect + devops + backend-dev + qa-critic (فيتو)

---

## السياق

الورشات والوكلاء الخلفيون يموتون عند خروج عملية Claude المنسّقة. اكتُشف حيّاً ثلاث مرات خلال جلسة 2026-07-15. العرَض المرئي: شارة «Unknown liveness» (T-907).

اجتمعت لجنة تقنية 2026-07-15 لمراجعة الحل المقترح (ربط Workflow بـDurableTask). جميع الأعضاء وكلاء متزامنون (تجنباً لإعادة الدخول للمشكلة).

---

## الوقائع المؤصَّلة

### 1. الركيزة الدائمة جاهزة فعلاً
- **المكون:** `server/modules/workflow-supervisor/` يطلق `systemd-run --user` وحدة عابرة.
- **الحالة:** مبنيّة وخاضعة للاختبار بالكامل خلف علم `WORKFLOW_SUPERVISOR` (مطفأ).
- **الاختبارات:** 146/146 مرّة أخضر (T-819..T-823).
- **التفعيل:** حيّ منذ 2026-07-11.
- **الحقائق المُتحقَّقة:**
  - وحدة systemd `--user` مستقلة عن PM2 وعملية Claude (PID 960، parent=init، Linger=yes).
  - supervision مفعَّل ومستقرّ (MainPID 1009 → independent من supervisor parent).
  - **النتيجة:** systemd-run ينجو قطع SSH وخروج المنسّق ولا يمسّ الورشات المُطلَقة.

### 2. اعتراض أداة Workflow — مستحيل بنيوياً
- **التاريخ:** Layer 2 مُستحدَث 2026-06-12 (commit 4bf9af31)، ثم أُبطل بـADR-053.
- **الحقيقة:** bypassPermissions يخفّي canUseTool؛ الحقن في نقطة الصدّة downstream من موت العملية.
- **الخلاصة:** لا يحمي من تيتّم.

### 3. السبب الجذري — طوبولوجيا العملية لا إهمال الكود
- **التسلسل الفعلي:**
  1. نسّاج يمرّر الـprompt نصّاً (input غير stream).
  2. SDK يستنتج `isSingleUserTurn=true` (اشتقاق سياق صحيح للـ CLI).
  3. SDK يُغلق stdin بعد أول نتيجة → انتهاء I/O.
  4. عملية Claude تخرج → جميع الأبناء يموتون (treekill في PM2 لا استثناء).
- **المرجع:** `docs/plans/B-103-DECISIONS-T827-T829-2026-07-10.md:167-171`
- **التداعي:** هذا الموت **متعامد تماماً على نموذج DurableTask** — لا المُغذّي ولا إهمال inline يقلّصانه.

### 4. تقنين مسارات البثّ الحيّ — ادّعاء مرفوض
- **الحقيقة:** systemd.ts:314 يسجّل `--output-format json` فقط (خزن، لا بثّ لحظي).
- **الواقع:** journal بلا timestamp دقيق، detach-and-tail غير موجود في البناء.
- **الخلاصة:** ادّعاء البثّ الحيّ مُسقَط؛ أقصى المتاح: بطاقة حالة non-LLM.

## القرار

### الحكم الأساسي
**المقترح الأصلي «ربط Workflow بـDurableTask يحلّ التيتّم» مرفوض كتأطير.**

**السبب:** المشكلة طوبولوجية (طبيعة CLI + SDK decision)، وليس نقصاً في دوام الخزن. حلّ خادمي لا يخاطب السبب الجذري.

### تقسيم الحلول — حسب السطح المعالجَ

#### سطح 1: ورشات واجهة نسّاج الصريحة
- **الحل:** الإطلاق الدائم (systemd-run) مستقبلاً.
- **القيد:** مؤجَّل خلف بوابة **T-714** (loss-meter الكمية) — لا تخمين، لا قياس.
- **الحالة:** يبقى معطّلاً حتى حكم المالك يفتح البوابة.

#### سطح 2: منسّق Claude Code CLI (حالة المستخدم)
- **الواقع:** خارج مدى أي حلّ خادمي **بنيوياً**.
- **tmux:** لا يحمي من خروج العملية عند حدّ السياق — يحمي من قطع SSH فقط.
- **الخلاصة:** هذا الألم **غير حلول** من الخادم. ينطبق الانضباط المتزامن أدناه.

### الحلول الفعّالة (معتمدة الآن)

#### 1. الانضباط المتزامن (Concurrent Pattern)
- **المبدأ:** وكلاء Agent خارج مظلة `wf-*`.
- **الإثبات الحيّ:** مهام كبيرة أُنجزت بنجاح 2026-06-30 و2026-07-02 و2026-07-09 و2026-07-15 بتقسيم موجاتٍ متزامنة.
- **الفائدة:** ينجو داخل الدور الحيّ للمنسّق.
- **المسار العملي المعتمد:** للعمل الطويل حقّاً، قسّمه موجاتٍ متزامنة.

#### 2. المخفِّف الفوري — Escape Hatch (متاح الآن بلا بناء)
لأي مهمة تتجاوز حدود جلسة المنسّق:

```bash
systemd-run --user --unit=wf-adhoc-$(date +%s) \
  -p CollectMode=inactive-or-failed \
  -p MemoryMax=2G \
  -p RuntimeMaxSec=7500 \
  --working-directory=/home/nassaj/Project/nassaj-dev \
  claude -p "<نص_المهمة>" --output-format=stream-json
```

- **الحماية:** ينجو من خروج Claude، قطع SSH، restart التطبيق.
- **السياق:** محفوظ في مجلّد home المستخدم (XDG_RUNTIME_DIR).
- **التنظيف:** `CollectMode=inactive-or-failed` يحذف الملفات بانتهاء الوحدة.

#### 3. حارس نقطة التفريخ
- **المسؤول:** T-829
- **الشرط:** منع إطلاق ورشات بخطأ قبل حكم T-714.

## شروط qa-critic الحاجبة

إذا أُحيي أي بناء تسليم لاحقاً (سطح 1 أو تحسين P2):

1. **إعادة تأطير الحكم:** لا «حلّ تيتّم» بل «تسليم مهام خلفية صريحة لسطح المنتج».
2. **فصل وإسقاط:** detach-and-tail منفصل؛ ادّعاء البثّ الحيّ مُسقَط.
3. **البوابة الكمية:** إخضاع P2/P3 لحكم T-714 قبل أي كود.
4. **برهان حصرية:** P2 لا يعيد تنفيذ B-129.
5. **توثيق التكافؤ:** byte-identical للمُغذّي المكتوب (أن وُجد).
6. **التوفيق:** الحزمة T-873 المتأخّرة تُسقَط من الشرط أو تُؤخَّر معاً.

## العواقب

### إيجابياً
- تجنّبنا بناء خطّ تسليم معقّد لمشكلة بنيوية الجذور.
- الركيزة المبنيّة جاهزة خلف علمها للمستقبل بلا بناء إضافي.
- المالك يملك escape hatch فوريّاً بلا انتظار.

### سلباً
- **ألم المنسّق يبقى مُدَاراً بالانضباط، لا محلولاً جذرياً** (الطبيعة الطوبولوجية تمنع حلاً خادمياً).
- خطة تشغيل devops (CollectMode، TasksMax، CPUQuota في systemd.ts؛ حارس علم حول exit 3 في safe-restart؛ تقنين linger+reaper في provision-node.sh) تبقى تحسينات قائمة **إن** فُعّلت الركيزة لاحقاً.
