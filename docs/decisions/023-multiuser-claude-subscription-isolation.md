## Context

دراسة جدوى (2026-06-08) لإعادة تعدّد المستخدمين على `nassaj-dev` بقيد **BYO اشتراك Claude لكل مستخدم** (OAuth Pro/Max، فوترة على حساب كل مستخدم) + ملفات/مشاريع/تعليمات مشتركة بالكامل + جلسات متزامنة.

**الحكم:** ممكن بتحفّظات — **البنية مبنية ~80% فعلاً ومُعطَّلة بالإعداد، لا إعادة معمار.**

الحقائق الحاملة التي تؤطّر القرار:

| الحقيقة | الدليل |
|---|---|
| **مسار chat/SDK سليم** — التزامن محلول | `query()` تُشغّل عملية `claude` CLI **فرعية حقيقية** بـ env خاص لكل استدعاء → `CLAUDE_CONFIG_DIR` لكل ابن = اعتماد per-user = فوترة متزامنة صحيحة. سلسلة الحقن: `resolve-provider-env.js:49,66` → `claude-sdk.js:179,784` → `chat-websocket.service.ts:218` (`ws.userId` من JWT). اعتراض «in-process global credential» سقط أمام الكود. |
| **العطل في مسار PTY فقط** | الطرفية (PTY) لا تمرّر `userId` ولا تحقن env المعزول، فغير-المالك لا يستطيع تسجيل اشتراكه عبرها، ومفتاح جلسة PTY بلا `userId` يسمح باختطاف الجلسة. (تفصيل في القرار 4 وعناصر م1 بـPHASE-MU.) |
| **«العزل» = نسبة فوترة فقط، ليس حدّاً أمنياً** | كل المستخدمين على **uid نظامي واحد** + نظام ملفات + مخزن محادثات مشترك. التأطير الأمين: «فريق متبادل الثقة، الخاصية الوحيدة per-user هي الفوترة». |

**تصحيحان من المراجعة المعمارية (موثّقان هنا):**

1. **الافتراضي = `isolated` لا `shared`:** الكود الافتراضي فعلاً `provider-sharing.js:38 → claude:'isolated'`. أي تعطيل قائم يأتي من **إعداد admin مخزَّن** (DB)، لا من الكود. تُؤكَّد القيمة الفعّالة **وقت التشغيل** في بوابة-0 (لا يُفترض من الكود).
2. **إصدار SDK = `@anthropic-ai/claude-agent-sdk@0.3.152`** (لا 0.2.116). يُثبَّت بدقّة في بوابة-0 ويُعاد التحقق عند أي ترقية.

## Decisions

### Decision 1 — المسار: BYO OAuth Pro/Max لكل مستخدم

اعتماد **اشتراك Claude OAuth Pro/Max لكل مستخدم** (BYO)، الفوترة على حساب كل مستخدم.

- **«العزل» = نسبة فوترة فقط، ليس حدّاً أمنياً** (كل المستخدمين على uid واحد، ثقة متبادلة).
- **مخاطرة ToS مقبولة للاستخدام الداخلي:** تشغيل `claude` برمجياً عبر خدمة متعددة المستخدمين دائمة هو الشكل الذي توجّهه Anthropic نحو `ANTHROPIC_API_KEY` → منطقة رمادية.
- **التحقق القانوني (ToS) مسار متوازٍ — حاجب للإطلاق العلني فقط**، غير حاجب للعمل الداخلي على `:3004`. البديل المُزيل للرمادي عند الحاجة: API key per-user (نفس السباكة، يغيّر نموذج الفوترة فقط).

#### تنقيح 1.1 — مسار اعتماد ثنائي حسب نوع الاستخدام (محسوم 2026-06-09 بعد بوابة-0 وتحقق قانوني)

التحقق القانوني المتوازي اكتمل وحسم الرمادية إلى **مسارين منفصلين بحسب نوع الاستخدام**، يُثبَّتان هنا كامتداد صريح للقرار 1 (لا يلغيانه):

| نوع الاستخدام | المسار المعتمد | الحالة |
|---|---|---|
| **داخلي** (فريق نسّاج، ثقة متبادلة) | **BYO OAuth** اشتراك Pro/Max لكل مستخدم | ✅ مقبول — يمضي فوراً |
| **علني / تجاري** | **محظور على OAuth-consumer.** المسار المتوافق الوحيد = **BYO-API-key تحت Commercial Terms** | 🚫 محجوب كبوابة في الكود |

- **الأساس القانوني للحظر العلني:** **Consumer Terms** (سارية 8 أكتوبر 2025) **§3(7)** يحظر الوصول الآلي إلا عبر Anthropic API Key، و**§2** يحظر إتاحة الحساب لطرف وسيط. لذا أي إطلاق علني على OAuth-consumer **غير متوافق**.
- **المسار العلني المتوافق الوحيد:** BYO-API-key تحت **Commercial Terms** — **نفس سباكة العزل**، يتغيّر **نموذج الفوترة فقط**.
- **التبرير المعماري (من architect):** طبقة العزل (`resolveProviderEnv`) **credential-type-agnostic** — تحقن **دليل اعتماد معزول per-user** لا توكناً بعينه؛ OAuth وAPI-key كلاهما يُحَل داخل نفس الدليل عبر سلسلة الأولوية في `claude-auth.provider.ts:106-129`. لذا التحول لـAPI-key لاحقاً **امتداد للسباكة لا إعادة معمار** — لا إهدار.
- **بوابة (في الكود لا سياسة):** مسار OAuth-consumer **يُحجب من الإطلاق العلني كبوابة في الكود**. الحسم المالي لنموذج الإطلاق يُحال لـ**business-advisor** حين يقترب الإطلاق. **إعادة فحص قانوني بالنسخة السارية وقت الإطلاق إلزامية.**

### Decision 2 — تصليب OS: «وثّق واقبل المخاطرة»

اعتماد **uid نظامي واحد** + صلاحيات ملفات **0600/0700** + **audit log**. **لا uid منفصل ولا container لكل مستخدم.**

- **السبب:** uid/container منفصل يتعارض مع **المشاركة الكاملة** للملفات/المحادثات ويضيف **عبء ذاكرة**.
- **القرار = «وثّق واقبل»:** خطر «سرقة اشتراك» (توكن قابل للقراءة تحت uid مشترك) **مقبول صراحةً** ضمن نموذج الفريق متبادل الثقة، ويُحصَّن بصلاحيات 0600/0700 + audit، ولاحقاً بتشفير التوكن في DB (انظر القرار 4).

### Decision 3 — MCP / الأدوات / الملفات: مشتركة بالكامل

**MCP servers والأدوات والملفات والمحادثات والتعليمات مشتركة بالكامل.** **فقط اشتراكات النماذج (الاعتماد per-user) معزولة.**

- **السبب:** المشاركة الحيّة الكاملة هي الغرض من البنية (Phase-MU، ADR-014). عزل MCP يناقض الغرض بلا فائدة أمنية تحت uid مشترك.

### Decision 4 — Onboarding: PTY مُصلَّح لـV1 ← ثم توكن مشفّر في DB

تسجيل اشتراك المستخدم في V1 عبر **PTY مُصلَّح** (سدّ ثغرتي م1)، ثم **`CLAUDE_CODE_OAUTH_TOKEN` مشفّر في DB** كتحصين تالٍ.

- **السبب:** سدّ ثغرتي PTY يفتح مسار تسجيل الاشتراك لغير-المالك (تمرير `userId` + حقن env + إدراج `userId` في مفتاح الجلسة).
- التوكن المشفّر في DB **يعزل تخزين الاشتراك فعلاً** تحت uid مشترك، فيخفّف مخاطرة القرار 2 (سرقة الاشتراك من نظام الملفات).

## Consequences

- **إيجابي:** لا إعادة معمار. مسار chat/SDK جاهز؛ تفعيل العزل = قلب إعداد admin مخزَّن (بلا restart). العمل المتبقّي مركَّز في مسار PTY + الحوكمة.
- **مقبول صراحةً (موثّق):** uid مشترك = لا حدّ أمني بين المستخدمين؛ توكن قابل للقراءة محلياً؛ مخاطرة ToS رمادية للاستخدام الداخلي.
- **حاجب للإطلاق العلني (لا للعمل الداخلي):** إغلاق ToS قانونياً (Decision 1) + موافقة PDPL مكتوبة.
- **قيد التشغيل (بوابة-0):** يُؤكَّد وقت التشغيل: غياب أي `ANTHROPIC_API_KEY`/`ANTHROPIC_AUTH_TOKEN`/`CLAUDE_CODE_OAUTH_TOKEN` **عام** يطغى عبر `claude-auth.provider.ts:109`، والقيمة الفعّالة لـ`provider-sharing`، وتثبيت SDK 0.3.152.
- **تبعية حادثة OOM:** مراقبة RSS مقابل `max_memory_restart` (تعدّد العمليات الفرعية يرفع الذاكرة) — وعي ضمن B-MU-SYNC.

> **تنبيه ملفّي:** `ecosystem.config.cjs` و`docs/upstream-contribution/` **لا تُلمس** في هذا التوثيق. أي قيم env تُسلَّم للمستخدم لتطبيقها بنفسه. جرد env الحيّ يجري في بوابة-0 (B-MU-G0-ENVINV) كقراءة فقط.
