# ADR-052 — رؤية محتوى الجلسة على مستوى المشروع

- **الحالة:** Accepted (اعتمد المالك 2026-06-30؛ إجماع architect + qa-critic)
- **التاريخ:** 2026-06-30
- **الصلة:** تنقيح B-105/B-106 (366c8452/6d509f78)، B-111 (f850b5e9)؛ تعدد المستخدمين (ADR-023/024)

---

## Context

نسّاج أداة تطوير داخلية لفريق موثوق واحد (مستأجر واحد). التخزين (JSONL) مشترك فيزيائياً،
والافتراض الاجتماعي `public = مقروء للفريق`.

**طبقة القائمة** (`getVisibleProjectPaths`) تكشف كل مشروع `public` أو يكون المستخدم عضواً أو مُشاركاً
فيه. **طبقة المحتوى** (جلب الجلسات والبحث) كانت محصورة على `participant-only` فقط (تنقيح B-105/B-106).

**المشكلة:** تناقض في وحدة المشاركة:
- الجلسة **تظهر** في قائمة المشاريع (`public` → شيئة) لأن مشروعها `public`.
- محتوى نفس الجلسة **غير مقروء** لنفس المستخدم (ليس مُشارك) → 404 أو حجب.

النتيجة: تنقيح B-105/B-106 كان صحيحاً (IDOR محمي: غير-العضو ⇒ 404)، لكنه فضح **عدم متسق** في
التصميم: **طبقة القائمة تقول «المشروع مرئي» بينما طبقة المحتوى تقول «الجلسة مخفية»**.

---

## Alternatives

1. **مشاركة صريحة per-session** — العضو يُشارك تصريحياً جلسة واحدة.
   - مرفوض: تعقيد الواجهة، يخالف افتراض الأداة (فريق موثوق، جلسات خاصة بالمشروع).

2. **تأجيل** — إبقاء التناقض، حجب محادثات فريق على مشاريع `public`.
   - مرفوض: يعطّل الغرض من الأداة (رؤية عمل الفريق مباشرة).

3. **رؤية على مستوى المشروع** (مختار) — توسيع طبقة المحتوى لتقبل **نفس معايير** القائمة:
   إذا رأى المستخدم المشروع (عامّ أو عضو أو مُشارك)، يرى محتوى جلساته.

---

## Decision

**استخراج `isProjectPathVisibleToUser`** من `getVisibleProjectPaths` كإسقاط أحادي-الصف
(primitive جديد)، وتطبيقه كفرع OR ثانٍ في طبقة قراءة المحتوى (`fetchHistory` و `filterSessionsByOwnership`).

### Primitive جديد: `isProjectPathVisibleToUser`

```typescript
// server/modules/database/repositories/projects.db.ts
isProjectPathVisibleToUser(projectPath: string, userId: number): boolean
```

**يعيد `true` إن كانت:**
- المشروع **موجود وغير مؤرشف** (`isArchived = 0`) بهذا المسار.
- **و** أحد الشروط:
  1. المشروع **عامّ** (`visibility = 'public'`).
  2. **أو** المستخدم **مُنشِئ** (`created_by = userId`).
  3. **أو** المستخدم **عضو** (في `project_members`).
  4. **أو** المستخدم **مُشارك بجلسة** (في `session_participants`).

**يعيد `false` إن كانت:**
- المسار غير صالح أو فارغ.
- لا توجد صفّ project بهذا المسار أو أُرشِّفت.
- المشروع خاص والمستخدم لا يستوفي أي شرط من الأربعة.
- `userId` **ليس عدداً صحيحاً** (مجهول / غير محقَّق) — **الحارس الـfail-closed يعمل قبل الفحص العامّ**
  فحتى مشروع عامّ **لا** يُبلَّغ مرئي لمُتصل مجهول.

### التطبيق

**في طبقة قراءة المحتوى، ثلاث نقاط تعديل:**

1. **`fetchHistory`** (`sessions.service.ts`):
   ```typescript
   if (!isProjectPathVisibleToUser(session.project_path, userId) && !isParticipant) {
     throw 403; // أو 404
   }
   ```

2. **`filterSessionsByOwnership`** (search service):
   ```typescript
   const rows = db.query(...)
   const visible = rows.filter(row =>
     isProjectPathVisibleToUser(row.project_path, userId) ||
     row.created_by === userId  // participant
   );
   ```

3. **`filterSessionsBySearchPhrase`** (search service):
   نفس الفلترة أعلاه — لا بحث على محادثات غير مرئية.

**لا حذف لفرع `isParticipant`:** يبقى كـ OR أول (أسرع، وحالات حدّية مثل جلسة orphan).

### الافتراض العام يبقى

لا migration: كل جلسة موجودة بالفعل في مشروع. `visibility` الحالي يظلّ ساري المفعول.
الـprimitive يُقيّم المشروع المُنشأ، لا المشروع المُفترَض.

---

## Consequences

### إيجابي
- **توحيد المعيار:** القائمة والمحتوى يستخدمان نفس الـpredicate → لا تناقض.
- **IDOR محمي:** مشروع خاص + غير-عضو ⇒ 404 (تنقيح B-105/B-106 محفوظ).
- **مرونة معقولة:** مشروع عامّ = فريق يرى محادثاته (الافتراض المنشود).
- **صفر migration:** لا تغيير بيانات.
- **fail-closed:** أي مُتصِل مجهول (غير عدد صحيح) مرفوض قبل أي فحص.

### مخاطر
- **اقتران بـqueries المشروع:** أي خطأ في DB queries نموذج عضوية المشروع ينعكس على المحتوى.
  - **التخفيف:** اختبارات ملكية قائمة (17/17 اختبار) + typecheck.
- **توسيع السطح من participant إلى «من يرى المشروع»:** يزيد من يقرأ.
  - **التخفيف:** القائمة تكشف المشروع **أصلاً** والتخزين مشترك والفريق موثوق.
    خارج نموذج المستأجر-الواحد يجب إعادة النظر.
- **مُشاركات لاحقة:** إن مشروع بدأ خاص ثم صار عامّاً، جلساته تصبح مرئية فوراً.
  - **بالتصميم:** متسق مع قرار visibility الديناميكي.

### العقد القادمة

- **Frontend:** إن كانت الواجهة تُعرّف رسائل 404 على محتوى، قد يُحسّن معاملة الحالة
  (مثل عرض «جلسة مخفية» بدل خطأ عام).
- **audit/logs:** تسجيل قراءات crosswise (مشروع عامّ محتوى فريق) قد يُمنِّح مرئية تمرير موثوقية.

## علاقته بـ B-105 و B-106

**لا نقض:** ADR-052 تنقيح (refinement) لا استرجاع:
- B-105 (366c8452): حظر قراءة رسائل جلسة على غير-العضو **في نفس الجلسة الخاصة**.
  - محفوظ: جلسة في مشروع خاص + غير-عضو = 403/404.
- B-106 (6d509f78): حظر بحث عبر جلسات المستخدم الآخر.
  - محفوظ: بحث على غير-جلساتك الخاصة بك يعود خالياً.
- **ADR-052:** توسيع **نطاق من يقرأ** من «مُشارك مباشر» إلى «من يرى المشروع الحاوي».
  - لا يُلغي الحدود الأمنية; يوضّحها فقط.
