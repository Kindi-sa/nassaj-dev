# Phase MU — Multi-User ⏳

**الحالة:** معلّقة — مخطّطة 2026-05-26 · **طبقة التفعيل V1 معتمدة 2026-06-08 (ADR-023)** · **شريحة B-MU-UX مكتملة + دفعة UI/إصلاحات 2026-06-10 (انظر قسم الدفعة أدناه)**
**المسؤول:** backend-dev + frontend-dev (architect للقرارات، scribe للتوثيق)
**القرارات المرجعية:** ADR-014, ADR-015, ADR-016, ADR-017 (في `~/.claude/alkindy/decisions/`) · **ADR-023** (`docs/decisions/023-multiuser-claude-subscription-isolation.md`) · **ADR-024** (`docs/decisions/024-multiuser-agy-subscription-isolation.md` — عزل اشتراك agy)

> **الهدف:** تحويل `nassaj-dev` إلى multi-user لفريق نسَّاج: عزل اعتماد per-user
> (Claude إلزامي، Gemini)، مع مشاركة حيّة كاملة للمحادثات والملفات والتعليمات، عبر
> Auth مدمجة داخل التطبيق وطبقة عزل مركزية `resolveProviderEnv`.

## طبقة التفعيل والتحصين (V1 — عزل الاشتراك)

> **مرجع القرار:** ADR-023 (Accepted 2026-06-08). هذه الطبقة **تُفعِّل** بنية Phase-MU
> القائمة أدناه (لا تُلغيها) وتُحصّنها. القرارات الأربعة محسومة؛ المتبقّي تنفيذ موجَّه.
>
> **الحقائق الحاملة:** البنية مبنية ~80%؛ **مسار chat/SDK سليم** (`claude-sdk.js:784`، عملية
> `claude` فرعية بـenv خاص لكل `userId`) فالتزامن محلول؛ **العطل في مسار PTY فقط**.
>
> **تصحيحان موثّقان:** (1) الافتراضي فعلاً `provider-sharing.js:38 → claude:'isolated'`
> (أي تعطيل من إعداد admin مخزَّن لا من الكود — يُؤكَّد وقت التشغيل في بوابة-0).
> (2) SDK = `@anthropic-ai/claude-agent-sdk@0.3.152` (يُثبَّت بدقّة).

### بوابة-0 — تأكيد وقت التشغيل (Gate 0)

#### B-MU-G0-TOS: قرار ToS موثّق
- **الوصف:** توثيق قرار مسار الاشتراك (BYO OAuth Pro/Max، Decision 1 من ADR-023) ومخاطرة ToS المقبولة للاستخدام الداخلي، مع تثبيت أن إغلاق ToS قانونياً **حاجب للإطلاق العلني فقط**.
- **معيار القبول:** القرار مكتوب ومُسجَّل (ADR-023 Decision 1)؛ تمييز صريح «داخلي مسموح / علني محجوب».
- **ملفات متأثّرة:** توثيق فقط (هذا الملف + ADR-023).
- **التبعيات:** لا شيء (نقطة البداية).

#### B-MU-G0-ENVINV: جرد env الحيّ
- **الوصف:** جرد متغيّرات البيئة الحيّة (قراءة فقط) لتأكيد **غياب** أي `ANTHROPIC_API_KEY` / `ANTHROPIC_AUTH_TOKEN` / `CLAUDE_CODE_OAUTH_TOKEN` **عام** يطغى على اعتمادات per-user عبر `claude-auth.provider.ts:109`.
- **معيار القبول:** تقرير جرد يثبت عدم وجود متغيّر عام طاغٍ؛ أي قيمة env تُسلَّم للمستخدم لتطبيقها بنفسه (لا يُعدَّل `ecosystem.config.cjs`).
- **ملفات متأثّرة:** قراءة فقط (env الحيّ، `claude-auth.provider.ts:109` مرجعاً).
- **التبعيات:** لا شيء.

#### B-MU-G0-DEFAULT: تأكيد القيمة الفعّالة لـprovider-sharing
- **الوصف:** تأكيد القيمة **الفعّالة وقت التشغيل** لـ`provider_sharing.claude` (الكود الافتراضي `isolated` في `provider-sharing.js:38`؛ التحقق من عدم وجود override من إعداد admin مخزَّن في DB).
- **معيار القبول:** يُثبَت أن القيمة الفعّالة معروفة (`isolated` أو override مخزَّن)؛ نقطة القلب محدَّدة (admin API، بلا restart).
- **ملفات متأثّرة:** قراءة `provider-sharing.js:38` + إعداد DB المخزَّن.
- **التبعيات:** لا شيء.

#### B-MU-G0-SDKPIN: تثبيت SDK 0.3.152
- **الوصف:** تثبيت `@anthropic-ai/claude-agent-sdk` على **0.3.152** بدقّة، وإعادة التحقق من سلسلة الحقن عند أي ترقية لاحقة.
- **معيار القبول:** الإصدار مثبَّت = 0.3.152؛ ملاحظة «أعد التحقق عند الترقية» موثّقة.
- **ملفات متأثّرة:** `package.json` / lockfile (تنفيذ لاحق بيد backend-dev — هنا توثيق المتطلَّب فقط).
- **التبعيات:** لا شيء.

### م1 — سدّ ثغرتي PTY (القلب)

#### B-MU-PTY-AUTH: تمرير request/userId لـhandleShellConnection
- **الوصف:** تمرير `request`/`userId` إلى `handleShellConnection` في `websocket-server.service.ts:67-68` ليطابق نمط مسار chat (السطر 73) الذي يمرّر `userId` بالفعل.
- **معيار القبول:** مسار PTY يستقبل `userId` الموثَّق من JWT تماماً كمسار chat؛ غير-المالك يستطيع بدء جلسة طرفية بهويّته.
- **ملفات متأثّرة:** `websocket-server.service.ts:67-68` (مرجعاً للسطر 73).
- **التبعيات:** بوابة-0.

#### B-MU-PTY-ENV: حقن resolveProviderEnv في env الطرفية
- **الوصف:** حقن مخرجات `resolveProviderEnv(userId, provider)` في env عملية الطرفية بـ`shell-websocket.service.ts:267-278` (حالياً بلا `CLAUDE_CONFIG_DIR` المعزول).
- **معيار القبول:** عملية الطرفية ترث env المعزول per-user (`CLAUDE_CONFIG_DIR` الصحيح)؛ تسجيل الاشتراك عبر PTY يكتب تحت اعتماد المستخدم لا المالك.
- **ملفات متأثّرة:** `shell-websocket.service.ts:267-278`.
- **التبعيات:** B-MU-PTY-AUTH.

#### B-MU-PTY-KEY: إدراج userId في ptySessionKey
- **الوصف:** إدراج `userId` في `ptySessionKey` (`shell-websocket.service.ts:201`) لمنع اختطاف جلسة PTY عبر المستخدمين.
- **معيار القبول:** مفتاح الجلسة فريد per-user؛ مستخدم لا يستطيع الوصول لجلسة PTY مستخدم آخر بنفس المفتاح.
- **ملفات متأثّرة:** `shell-websocket.service.ts:201`.
- **التبعيات:** B-MU-PTY-AUTH.

#### B-MU-PTY-TEST: اختبار ثغرتي PTY
- **الوصف:** اختبار يثبت أن PTY يحقن env المعزول الصحيح per-user وأن المفتاح يمنع الاختطاف عبر المستخدمين.
- **معيار القبول:** مستخدمان عبر PTY يكتبان تحت `CLAUDE_CONFIG_DIR` مختلف؛ محاولة وصول متقاطعة للجلسة مرفوضة.
- **ملفات متأثّرة:** اختبارات (suite الطرفية).
- **التبعيات:** B-MU-PTY-AUTH, B-MU-PTY-ENV, B-MU-PTY-KEY.

### م2 — تفعيل العزل

#### B-MU-ISO-FLIP: قلب الإعداد لـisolated
- **الوصف:** قلب القيمة الفعّالة لـ`provider_sharing.claude` إلى `isolated` عبر admin API (بلا restart)، بعد تأكيد بوابة-0.
- **معيار القبول:** القيمة الفعّالة = `isolated`؛ كل spawn لـClaude يبني env per-user عبر `resolveProviderEnv`.
- **ملفات متأثّرة:** إعداد admin المخزَّن (DB) — لا كود.
- **التبعيات:** بوابة-0, م1.

#### B-MU-ISO-E2E: اختبار E2E (= M-ISO-E2E)
- **الوصف:** **نفس** بند `M-ISO-E2E` القائم أدناه (مستخدمان باعتمادين مختلفين / نفس brain). يُعاد استخدامه كبوابة تفعيل V1.
- **معيار القبول:** كما في `M-ISO-E2E`: عزل اعتماد Claude فعلي + مشاركة محادثات + نفس brain + لا خلط تحت spawn متزامن.
- **ملفات متأثّرة:** اختبارات E2E.
- **التبعيات:** B-MU-ISO-FLIP, B-MU-PTY-TEST.

#### B-MU-ISO-AGYLOCK: قفل discovery ضيّق
- **الوصف:** = `B-ISO-AGYLOCK` القائم أدناه (قفل لحظي على نافذة اكتشاف brain UUID لمسار agy فقط).
- **معيار القبول:** كما في `B-ISO-AGYLOCK`: spawnان متزامنان لـagy لا يختلطان؛ القفل لا يحجب القراءات ولا المزوّدين الآخرين.
- **ملفات متأثّرة:** مسار بدء agy.
- **التبعيات:** B-ISO-RESOLVER (أدناه).

### م3 — حوكمة + تصليب

#### B-MU-OS-PERM: صلاحيات 0600/0700 + DB 0600
- **الوصف:** ضبط صلاحيات `~/.nassaj-users/<uid>/` على 0700 والملفات الحسّاسة (التوكن) على 0600، و`db.sqlite` (+WAL/SHM) على 0600.
- **معيار القبول:** الصلاحيات مطبَّقة ومتحقَّق منها؛ لا قارئ آخر على الجهاز خارج `nassaj`.
- **ملفات متأثّرة:** provisioning + DB (يدمج/يعزّز `m-DBPERM` أدناه).
- **التبعيات:** م2.

#### B-MU-GOV-OWNER: owner واحد
- **الوصف:** تثبيت أن النظام يملك **owner واحداً** يدير المالك المشترك لـagy والحوكمة.
- **معيار القبول:** bootstrap ينشئ owner واحداً لا يُكرَّر؛ لا تعدّد owners.
- **ملفات متأثّرة:** مسار bootstrap (يستند لـC-AUTH-2 أدناه).
- **التبعيات:** م2.

#### B-MU-GOV-RATELIMIT: rate limit
- **الوصف:** = `m-RATELIMIT` القائم أدناه (rate limit على endpoints الـauth).
- **معيار القبول:** كما في `m-RATELIMIT`: تجاوز الحد يردّ 429؛ الاستخدام الطبيعي غير متأثّر.
- **ملفات متأثّرة:** endpoints الـauth.
- **التبعيات:** م2.

> **B-MU-OS-HARDEN — محسوم (Decision 2):** «وثّق واقبل المخاطرة» — uid نظامي واحد + 0600/0700 + audit. **لا uid/container منفصل** (يتعارض مع المشاركة الكاملة ويضيف عبء ذاكرة). لا عمل تنفيذي إضافي عدا B-MU-OS-PERM.

### م4 — onboarding + تزامن + امتثال

#### B-MU-ONBOARD: تسجيل الاشتراك
- **الوصف:** تسجيل اشتراك المستخدم عبر **PTY مُصلَّح لـV1** (بعد م1)، ثم **`CLAUDE_CODE_OAUTH_TOKEN` مشفّر في DB** كتحصين تالٍ (Decision 4).
- **معيار القبول:** مستخدم جديد يسجّل اشتراكه عبر PTY بهويّته؛ مسار التوكن المشفّر في DB موثّق كخطوة تالية.
- **ملفات متأثّرة:** مسار onboarding + (لاحقاً) تخزين DB مشفّر.
- **التبعيات:** م1, م2.

#### B-MU-MCP: MCP مشترك
- **الوصف:** = **محسوم (Decision 3):** MCP/الأدوات/الملفات **مشتركة بالكامل**؛ فقط اشتراكات النماذج معزولة. لا عمل عزل MCP.
- **معيار القبول:** MCP يعمل مشتركاً دون كسر؛ بند مسجَّل كقرار لا كعمل تنفيذي.
- **ملفات متأثّرة:** لا شيء (قرار).
- **التبعيات:** م2.

#### B-MU-SYNC: تحقق تزامن + مراقبة RSS
- **الوصف:** تحقق سلامة التزامن (مشاركة حيّة + قفل discovery) + **مراقبة RSS مقابل `max_memory_restart`** (تعدّد العمليات الفرعية يرفع الذاكرة) مع **وعي حادثة OOM**.
- **معيار القبول:** التزامن سليم تحت حمل متعدّد المستخدمين؛ RSS مرصود مقابل سقف الذاكرة؛ سيناريو OOM موثّق.
- **ملفات متأثّرة:** مراقبة فقط (لا يُعدَّل `ecosystem.config.cjs` — قيم تُسلَّم للمستخدم).
- **التبعيات:** م2.

#### B-MU-PDPL: موافقة PDPL مكتوبة
- **الوصف:** موافقة PDPL مكتوبة على معالجة بيانات المستخدمين. **حاجب للإطلاق.**
- **معيار القبول:** موافقة موقّعة قبل أي إطلاق علني.
- **ملفات متأثّرة:** توثيق امتثال.
- **التبعيات:** م2.

#### B-MU-RUNBOOK: runbook
- **الوصف:** = `m-RUNBOOK` القائم أدناه (runbook ثنائي اللغة لـbootstrap/invite/provision/migration) مع إضافة خطوات تسجيل الاشتراك per-user.
- **معيار القبول:** كما في `m-RUNBOOK` + تغطية onboarding الاشتراك.
- **ملفات متأثّرة:** runbook.
- **التبعيات:** كل بنود م1-م3.

#### B-MU-LEGAL: إغلاق ToS
- **الوصف:** إغلاق التحقق القانوني لـToS اشتراكات Claude (Decision 1). **حاجب للإطلاق العلني فقط.**
- **معيار القبول:** قرار قانوني مكتوب (BYO OAuth مقبول، أو التحوّل لـAPI key per-user) قبل أي إطلاق علني.
- **ملفات متأثّرة:** توثيق قانوني/قرار.
- **التبعيات:** B-MU-G0-TOS.

### مسار agy — عزل اشتراك Antigravity (AGY-*)

> **مرجع القرار:** ADR-024 (Accepted 2026-06-09). يوسّع عزل الاشتراك per-user من Claude إلى
> مزوّد **`agy` (Antigravity CLI)** — **محور هذا الـfork**. يختلف معمارياً عن Claude:
> لا knob اعتماد مخصّص (نظير `CLAUDE_CONFIG_DIR`)، بل **طغيان `HOME` بالكامل**، والاعتماد في
> `~/.gemini/antigravity-cli/antigravity-oauth-token`. الآلية القديمة افترضت كفاية طغيان HOME
> لكن `provisionUserDirs` **لم يعالج `agy` إطلاقاً** → القلب وحده كان **يكسر المالك**.
> هذا المسار يعتمد على **إصلاح م1/PTY المُنجَز** (لا يكرّره).

#### B-AGY-BRAIN: brain مشترك عبر symlink (Decision 1)
- **الوصف:** symlink لـbrain agy المشترك داخل `HOME` المعزول لكل مستخدم — نظير `projects/` في Claude (`B-ISO-CLAUDE`). المعزول = الاشتراك/الفوترة فقط؛ المحادثات/brain مشتركة.
- **معيار القبول:** مستخدمان مختلفان يريان نفس brain/محادثات agy عبر symlink؛ لا عزل للمحادثات.
- **ملفات متأثّرة:** مسار provisioning لـagy (symlink brain).
- **التبعيات:** B-AGY-PROVISION.

#### B-AGY-OWNER-EXEMPT: إعفاء المالك بـsymlink التوكن (Decision 2)
- **الوصف:** symlink لتوكن المالك (`~/.gemini/antigravity-cli/antigravity-oauth-token`) داخل دليله المعزول — مرآة إعفاء المالك في Claude — كي لا يفقد المالك اعتماد agy عند قلب التبديل.
- **معيار القبول:** بعد القلب لـ`isolated`، المالك يحتفظ باعتماد agy ويشغّله دون كسر؛ يُتحقَّق صراحةً كبوابة قبل اعتبار القلب ناجحاً.
- **ملفات متأثّرة:** مسار provisioning للمالك (symlink التوكن).
- **التبعيات:** B-AGY-PROVISION.

#### B-AGY-PROVISION: معالجة agy في provisionUserDirs (Decision 3)
- **الوصف:** إضافة فرع `agy` إلى `provisionUserDirs` (كان يتجاهل agy إطلاقاً): إنشاء دليل agy المعزول داخل `HOME` المعزول + صلاحيات `0700` للدليل و`0600` للتوكن (نظير `B-MU-OS-PERM`).
- **معيار القبول:** قبول دعوة يُنشئ دليل agy المعزول للمستخدم بصلاحيات مقيّدة؛ المالك وغير-المالك كلاهما مُعالَج (لا أحد يُترك بلا مسار).
- **ملفات متأثّرة:** `provisionUserDirs` (يدمج/يعزّز نمط `B-ISO-PROVISION`).
- **التبعيات:** B-MU-PTY-AUTH/ENV (م1 المُنجَز), C-AUTH-4.

#### B-AGY-STATUS: endpoint حالة agy (Decision 3)
- **الوصف:** endpoint لحالة اعتماد agy للمستخدم (نظير endpoint حالة Claude) — لتشخيص الاعتماد ودعم onboarding.
- **معيار القبول:** endpoint يرجع حالة اعتماد agy per-user (مسجَّل/غير مسجَّل)؛ يُستهلَك من واجهة onboarding.
- **ملفات متأثّرة:** مسار API الحالة.
- **التبعيات:** B-AGY-PROVISION.

#### B-AGY-PTY: فرع agy في حقن env الطرفية (Decision 3)
- **الوصف:** فرع `agy` في حقن env المعزول داخل PTY (طغيان `HOME` المعزول للمستخدم) — يعتمد على **إصلاح م1/PTY المُنجَز** (`B-MU-PTY-AUTH`/`B-MU-PTY-ENV`)، لا يكرّره.
- **معيار القبول:** عملية الطرفية ترث `HOME` المعزول الصحيح لـagy per-user؛ تسجيل اشتراك agy عبر PTY يكتب تحت اعتماد المستخدم لا المالك.
- **ملفات متأثّرة:** فرع agy في مسار حقن env الطرفية (نظير `shell-websocket.service.ts`).
- **التبعيات:** B-MU-PTY-ENV (مُنجَز), B-AGY-PROVISION.

#### B-AGY-ONBOARD: واجهة onboarding تسجيل agy (Decision 3)
- **الوصف:** واجهة تسجيل اشتراك agy لغير-المالك (عبر PTY المُصلَّح + فرع agy)، نظير onboarding Claude.
- **معيار القبول:** مستخدم غير-مالك يسجّل اشتراك agy بهويّته عبر الواجهة؛ يُتحقَّق من الكتابة تحت اعتماده.
- **ملفات متأثّرة:** واجهة onboarding (فرع agy).
- **التبعيات:** B-AGY-PTY, B-AGY-STATUS.

#### B-AGY-FLIP: تفعيل عكوس مؤجَّل عبر admin API بعد restart (Decision 4)
- **الوصف:** كود agy يُشحن خاملاً (DB يبقى `agy = shared`)؛ القلب لـ`isolated` يتم عبر **admin API بعد restart** (بلا restart إضافي مخصّص لـagy)، **قابل للتراجع فوراً** بعد التحقق من إعفاء المالك. **لم يُضمّ إلى restart Claude الحالي** تفادياً لكسر agy.
- **معيار القبول:** عند الشحن السلوك بلا تغيير (`shared`)؛ القلب لـ`isolated` ممكن بلا restart إضافي؛ التراجع لـ`shared` فوري؛ B-AGY-OWNER-EXEMPT مُتحقَّق قبل القلب.
- **ملفات متأثّرة:** إعداد admin المخزَّن (DB) — لا كود سلوكي عند الشحن.
- **التبعيات:** B-AGY-PROVISION, B-AGY-OWNER-EXEMPT, B-AGY-PTY.

#### B-AGY-TOKENENC: تشفير توكن agy في DB (تحصين لاحق)
- **الوصف:** تشفير توكن agy في DB — نظير قرار التوكن المشفّر لـClaude (ADR-023 Decision 4) — لعزل تخزين الاشتراك فعلاً تحت uid مشترك. **تحصين تالٍ، لا حاجب للتفعيل الداخلي.**

### شريحة B-MU-UX — طبقة العزو والحضور (Attribution & Presence UX)

> **سياق التصميم:** مساحة العمل **مشتركة بالقصد** (إخوة، ثقة كاملة، وصول كامل للملفات بين
> المستخدمين). هذه الشريحة تضيف **رؤية/عزواً (visibility/attribution)** لا **عزلاً** — تجعل
> المساحة المشتركة **مقروءة**: مَن يملك كل محادثة، مَن يعمل الآن، وفلترة المستخدم لعمله. بيانات
> الملكية **موجودة فعلاً** في جدول `session_participants` (دور `owner` مقابل `participant`) لكنها
> **غير مستهلَكة من الواجهة**. تشمل أيضاً هويّة git per-user وإصلاحَي تزامن.
>
> **بوابة الشريحة:** أساس الـAPI (`B-MU-UX-API`) **يهبط أولاً** قبل بنود الواجهة (الشارة/الرسالة/الفلتر)
> التي تعتمد عليه. هويّة git (`B-MU-UX-GIT-ID`) وإصلاحَا التزامن (`-FIX-MIRROR`/`-FIX-WSROTATE`) مستقلّة.

  **أي** جلسات تُرجَع (تبقى كلها — المشاركة محفوظة)؛ فقط إثراء كل جلسة ببيانات الملكية.

  مشارك تتراجع برشاقة (`owner=null` أو owner مُمهَّد/bootstrap)؛ لا تغيير في مجموعة الجلسات المُرجَعة.
- **ملفات متأثّرة:** `server/modules/projects/services/projects-with-sessions-fetch.service.ts`،

| **B-MU-LEGAL (إغلاق ToS)** | تسجيل (مقبول داخلياً) | **🔴 حاجب** |
| **B-MU-PDPL (موافقة مكتوبة)** | تسجيل | **🔴 حاجب** |
| B-MU-OS-HARDEN / B-MU-MCP | قرار محسوم (لا عمل) | قرار محسوم |

## Auth (C-AUTH-*)

### C-AUTH-1: جداول `users` + `audit_log`
- **الوصف:** إضافة جدول `users` (id, username, password_hash[argon2id], role[owner/admin/user],
  created_at, status) وجدول `audit_log` (id, user_id, action, metadata, timestamp) داخل DB
  الموجودة `db.sqlite`. لا جدول `sessions` (JWT stateless).
- **التبعيات:** لا شيء (نقطة البداية).
- **معيار القبول:** migration يُنشئ الجدولين على DB قائمة دون فقد بيانات؛ schema snapshot test يمرّ.

### C-AUTH-2: JWT stateless + bootstrap owner
- **الوصف:** إصدار/تحقق JWT بلا حالة خادم (secret من `.env`). عند أول تشغيل على DB فارغة من
  المستخدمين، إنشاء owner تلقائياً (bootstrap)، لا يُكرَّر بعد وجود مستخدم.
- **التبعيات:** C-AUTH-1.
- **معيار القبول:** أول تشغيل يُنشئ owner واحداً؛ تشغيل لاحق لا يُكرّره؛ JWT صالح يُتحقَّق منه بلا قراءة DB للجلسة.

### C-AUTH-3: auth middleware (HTTP 401 + WS upgrade)
- **الوصف:** middleware يحرس كل طلبات HTTP المحمية (يردّ **401** بلا JWT صالح) ويحرس
  **WebSocket upgrade** (يرفض الاتصال بلا JWT صالح). يستخرج `userId` ويمرّره للطبقات الأدنى.
- **التبعيات:** C-AUTH-2.
- **معيار القبول:** طلب HTTP بلا/بـ JWT غير صالح → 401؛ WS upgrade بلا JWT صالح → رفض؛
  طلب صالح يمرّر `userId` الصحيح.

### C-AUTH-4: invite flow
- **الوصف:** إنشاء دعوة (owner/admin) → token دعوة → قبول الدعوة بإنشاء كلمة مرور →
  حساب `user`. لا تسجيل عام إطلاقاً (invite-only).
- **التبعيات:** C-AUTH-2.
- **معيار القبول:** دعوة → قبول → دخول يعمل E2E؛ محاولة تسجيل بلا دعوة مرفوضة.

## Isolation (B-ISO-*)

### B-ISO-RESOLVER: `resolveProviderEnv(userId, provider)`
- **الوصف:** الطبقة المحورية — **مصدر الحقيقة الوحيد** للعزل. تبني env المعزول لكل spawn
  حسب المزوّد وتُستهلَك من كل المزوّدين. تُرجِع المسارات/المتغيرات الصحيحة لكل من
  Claude/Gemini/Codex/agy.
- **التبعيات:** C-AUTH-3 (يوفّر `userId` الموثَّق).
- **معيار القبول:** كل spawn لأي مزوّد يمرّ عبر هذه الدالة لا غيرها؛ تُرجِع env مختلف
  per-user لـ Claude/Gemini و env المالك المشترك لـ agy.

### B-ISO-CLAUDE: `CLAUDE_CONFIG_DIR` + symlinks
- **الوصف:** ضبط `CLAUDE_CONFIG_DIR=~/.nassaj-users/<userId>/.claude` (اعتماد معزول)، مع
  symlink لـ `projects/` و `CLAUDE.md` و `NASSAJ.md` نحو الجذر المشترك (محادثات/تعليمات مشتركة).
- **التبعيات:** B-ISO-RESOLVER.
- **معيار القبول:** مستخدمان مختلفان لهما `CLAUDE_CONFIG_DIR` مختلف فعلياً؛ كلاهما يرى نفس
  `projects/` والتعليمات عبر symlink.

### B-ISO-GEMINI: `GEMINI_CLI_HOME`
- **الوصف:** عزل اعتماد Gemini عبر `GEMINI_CLI_HOME` لكل مستخدم (نمط قائم في
  `server/gemini-cli.js:83`)؛ المحادثات تبقى مشتركة.
- **التبعيات:** B-ISO-RESOLVER.
- **معيار القبول:** `GEMINI_CLI_HOME` مختلف per-user؛ المحادثات مشتركة بين المستخدمين.

### B-ISO-CODEX: عزل Codex (مؤجَّل)
- **الوصف:** عزل اعتماد Codex مؤجَّل (استخدام شبه معدوم). يبقى مشتركاً مؤقتاً، مع توثيق
  نقطة الإدراج في `resolveProviderEnv` لتفعيله لاحقاً.
- **التبعيات:** B-ISO-RESOLVER.
- **معيار القبول:** Codex يعمل كما هو دون كسر؛ بند مسجَّل كـ deferred لا blocking.

### B-ISO-PROVISION: provisioning `~/.nassaj-users/<uid>` + symlinks
- **الوصف:** عند **قبول الدعوة** (C-AUTH-4)، إنشاء `~/.nassaj-users/<userId>/.claude`
  (و `.gemini` حسب الحاجة) بصلاحيات مقيّدة + symlinks المحادثات/التعليمات تلقائياً.
- **التبعيات:** C-AUTH-4, B-ISO-CLAUDE, B-ISO-GEMINI.
- **معيار القبول:** قبول دعوة يُنشئ المجلد والـ symlinks تلقائياً؛ المستخدم الجديد يشغّل
  Claude باعتماده الخاص فوراً ويرى المحادثات المشتركة.

### B-ISO-AGYLOCK: قفل discovery ضيّق
- **الوصف:** قفل in-process ضيّق لحظي على **نافذة اكتشاف brain UUID** في مسار بدء agy فقط
  (لا قفل عام)، يُحرَّر فور إسناد الـ UUID — لتفادي خلط المحادثات تحت التزامن.
- **التبعيات:** B-ISO-RESOLVER.
- **معيار القبول:** spawnان متزامنان لـ agy لا يختلطان في إسناد brain UUID؛ القفل لا يحجب
  قراءات المحادثات ولا spawnات المزوّدين الآخرين.

## UI (C-UI-*)

### C-UI-1: تسجيل دخول RTL
- **الوصف:** شاشة تسجيل دخول (username + password) بدعم RTL عربي كامل.
- **التبعيات:** C-AUTH-2, C-AUTH-3.
- **معيار القبول:** الدخول يعمل؛ الشاشة RTL صحيحة بصرياً.

### C-UI-2: إدارة مستخدمين RBAC
- **الوصف:** لوحة إدارة المستخدمين (قائمة، أدوار، تعطيل/حذف) حسب الصلاحية
  (owner/admin يديران؛ user لا يرى اللوحة).
- **التبعيات:** C-AUTH-3.
- **معيار القبول:** owner/admin يديرون المستخدمين؛ user محجوب؛ الإجراءات تُسجَّل في audit_log.

### C-UI-3: دعوات
- **الوصف:** واجهة إنشاء دعوة وعرض الدعوات المعلّقة ونسخ رابط/token الدعوة.
- **التبعيات:** C-AUTH-4, C-UI-2.
- **معيار القبول:** owner/admin ينشئ دعوة ويحصل على رابط؛ الدعوة المقبولة تختفي من المعلّقة.

## Hardening / Cross-cutting (m-*) و E2E

### m-DBPERM: chmod 600 على DB
- **الوصف:** `chmod 600` على `db.sqlite` (+ WAL/SHM والنسخ الاحتياطية) ليقرأها `nassaj` حصراً.
- **التبعيات:** C-AUTH-1.
- **معيار القبول:** صلاحيات DB = 600؛ لا قارئ آخر على الجهاز.

### m-RATELIMIT: rate limit على auth
- **الوصف:** rate limiting على endpoints الـ auth (login، قبول الدعوة) لمنع brute force.
- **التبعيات:** C-AUTH-2, C-AUTH-4.
- **معيار القبول:** تجاوز الحد يردّ 429؛ الاستخدام الطبيعي غير متأثّر.

### m-RUNBOOK: runbook ثنائي اللغة
- **الوصف:** runbook (عربي سرد + إنجليزي للأوامر/الحقول) لـ bootstrap owner، إنشاء دعوة،
  provisioning، والترحيل للإنتاج ونقل `~/.nassaj-users/`.
- **التبعيات:** كل البنود التنفيذية أعلاه.
- **معيار القبول:** runbook يغطّي bootstrap + invite + provision + migration بخطوات قابلة للتنفيذ.

### M-ISO-E2E: اختبار مستخدمين باعتمادين مختلفين / نفس brain
- **الوصف:** اختبار E2E: مستخدمان عبر دعوتين، كلٌّ باعتماد Claude مختلف
  (`CLAUDE_CONFIG_DIR` منفصل)، يريان نفس المحادثات المشتركة ونفس brain agy دون خلط تحت التزامن.
- **التبعيات:** كل B-ISO-* و C-AUTH-* و C-UI-*.
- **معيار القبول:** الاختبار يثبت: عزل اعتماد Claude فعلي + مشاركة محادثات + نفس brain +
  لا خلط تحت spawn متزامن → بوابة المرحلة (GATE) تُجتاز.

## Critical Path

```
C-AUTH-1 → C-AUTH-2 → C-AUTH-3 → B-ISO-RESOLVER
                                      ├─▶ B-ISO-CLAUDE ─┐
                                      └─▶ B-ISO-GEMINI ─┤
                                                        ▼
                                              B-ISO-PROVISION
                                                        ▼
                                                  C-UI-1/2/3
                                                        ▼
                                                   M-ISO-E2E
                                                        ▼
                                                      GATE
```

- `B-ISO-CLAUDE ∥ B-ISO-GEMINI` متوازيان بعد `B-ISO-RESOLVER`.
- `C-AUTH-4` (invite) فرع من `C-AUTH-2`، شرط لـ `B-ISO-PROVISION` و `C-UI-3`.
- `B-ISO-CODEX` مؤجَّل (deferred، خارج المسار الحرج).
- `B-ISO-AGYLOCK` فرع من `B-ISO-RESOLVER`، يُتحقَّق منه في `M-ISO-E2E`.
- `m-*` (DBPERM/RATELIMIT/RUNBOOK) cross-cutting، تُغلَق قبل GATE.
